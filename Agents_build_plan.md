## 2. Authentication

Every request needs a Bearer token in the header:

```js
const PAWAPAY_API_TOKEN = "YOUR_SANDBOX_TOKEN"; // store in app config / env

const headers = {
  "Authorization": `Bearer ${PAWAPAY_API_TOKEN}`,
  "Content-Type": "application/json",
};
```

Get your sandbox token from the pawaPay Dashboard under API tokens. Do not commit this token to a public GitHub repo — even client-side, use an Expo env variable (`EXPO_PUBLIC_PAWAPAY_TOKEN`).

## 3. Core Concept — Everything is Asynchronous

Every pawaPay operation (deposit, payout, refund) works in two steps:

1. **You initiate** — you POST a request with a unique UUID you generate. pawaPay responds immediately with `ACCEPTED`, `REJECTED`, or `DUPLICATE_IGNORED`. This is NOT the final result — it just means pawaPay received your request.
2. **You find out the final result** — either by polling `GET` on a status endpoint, or via webhook callback (skip callbacks for the hackathon — just poll).

**Critical rule**: Always generate the UUID yourself BEFORE calling the API, and store it in your Supabase database immediately. This UUID (`depositId`, `payoutId`, or `refundId`) is how you match the transaction to your circle/wallet/escrow record later.

```js
import { v4 as uuidv4 } from 'uuid'; // or expo-crypto's randomUUID()

const depositId = uuidv4();
// Save depositId to Supabase against this contribution BEFORE calling pawaPay
```

## 4. Endpoint Reference

### 4.1 Initiate Deposit — Collect money FROM a user

**Use for**: wallet top-ups, circle contributions, escrow funding.

```
POST https://api.sandbox.pawapay.io/v2/deposits
```

```js
async function initiateDeposit({ depositId, phoneNumber, provider, amount, currency = "XAF", note }) {
  const response = await fetch("https://api.sandbox.pawapay.io/v2/deposits", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${PAWAPAY_API_TOKEN}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      depositId,
      payer: {
        type: "MMO",
        accountDetails: {
          phoneNumber, // digits only, no +, no leading zero, country code included e.g. "237653456789"
          provider,    // e.g. "MTN_MOMO_CMR" or "ORANGE_CMR"
        },
      },
      clientReferenceId: "CIRCLE-123", // optional — use your circle_id or wallet id here
      customerMessage: "MboaPay deposit", // 4-22 chars, alphanumeric + spaces only
      amount: String(amount), // MUST be a string, e.g. "5000"
      currency,
    }),
  });
  return await response.json();
  // { depositId, status: "ACCEPTED" | "REJECTED" | "DUPLICATE_IGNORED", created, failureReason? }
}
```

**Important field rules**:
- `amount` must be a STRING, not a number. No leading zeroes (except for values < 1).
- `phoneNumber` — digits only, no `+`, no leading `0`, must include country code. Example for Cameroon: `237653456789`.
- `customerMessage` — between 4 and 22 characters, only letters/numbers/spaces.
- `depositId` — must be a valid UUIDv4, generated by you, unique per request.

### 4.3 Initiate Refund — Return money to a user

**Use for**: member exits a circle before disbursement, escrow cancelled before recipient confirms.

```
POST https://api.sandbox.pawapay.io/v2/refunds
```

```js
async function initiateRefund({ refundId, depositId, amount, currency = "XAF" }) {
  const response = await fetch("https://api.sandbox.pawapay.io/v2/refunds", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${PAWAPAY_API_TOKEN}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      refundId,        // UUIDv4 you generate
      depositId,        // the ORIGINAL deposit's ID that you're refunding
      amount: String(amount), // omit this field entirely for a FULL refund; include it for PARTIAL
      currency,
    }),
  });
  return await response.json();
}
```

Refunds always reference the original `depositId`. You can issue multiple partial refunds as long as the total doesn't exceed the original deposit amount.

### 4.4 Check Deposit Status (Polling)

**Use this** instead of webhooks for the hackathon. Poll every 3-5 seconds after initiating a deposit until status is final.

```
GET https://api.sandbox.pawapay.io/v2/deposits/{depositId}
```

```js
async function checkDepositStatus(depositId) {
  const response = await fetch(`https://api.sandbox.pawapay.io/v2/deposits/${depositId}`, {
    headers: { "Authorization": `Bearer ${PAWAPAY_API_TOKEN}` },
  });
  return await response.json();
  // status: "ACCEPTED" | "SUBMITTED" | "COMPLETED" | "FAILED"
}

// Simple polling helper for React Native / Expo
async function pollDepositUntilFinal(depositId, { intervalMs = 3000, timeoutMs = 60000 } = {}) {
  const start = Date.now();
  while (Date.now() - start < timeoutMs) {
    const result = await checkDepositStatus(depositId);
    if (["COMPLETED", "FAILED"].includes(result.status)) return result;
    await new Promise(r => setTimeout(r, intervalMs));
  }
  throw new Error("Deposit status polling timed out");
}
```

## 7. Feature-to-Endpoint Mapping for MboaPay

### Wallet
| User action | Endpoint(s) |
|---|---|
| Top up wallet | `POST /v2/deposits` → poll `GET /v2/deposits/:id` → on COMPLETED, increment `wallets.balance` in Supabase |
| Auto-contribution deduction | No pawaPay call — this is purely internal. Deduct from `wallets.balance` in Supabase on schedule, then call `POST /v2/deposits` is NOT needed since the money is already in the wallet logically — just record it as a `contributions` row. Real money only moves pawaPay-side at top-up and at disbursement. |
| Withdraw from wallet | `POST /v2/payouts` → poll `GET /v2/payouts/:id` → on COMPLETED, decrement `wallets.balance` |

### Savings Circles (Solo & Pool)
| User action | Endpoint(s) |
|---|---|
| Member makes a contribution | `POST /v2/deposits` → poll until COMPLETED → insert row in `contributions` table |
| Circle target reached → auto-disburse | `POST /v2/payouts` to the member's own phone (Solo/Pool) → poll until COMPLETED → insert row in `disbursements` |
| Member exits early | `POST /v2/refunds` referencing their original `depositId` → poll until COMPLETED |

### Njangi Rotation
| User action | Endpoint(s) |
|---|---|
| Member joins, pays commitment deposit | `POST /v2/deposits` (small amount) → poll until COMPLETED → update `circle_members.deposit_status` |
| Round complete, pot disburses to recipient | `POST /v2/payouts` to that round's recipient phone → poll until COMPLETED → insert `disbursements` row, increment `rotation_order` pointer |
| Member defaults | No pawaPay call — internal logic locks the member and may trigger a partial use of their commitment deposit via ledger adjustment, not an actual refund/payout |

### Escrow
| User action | Endpoint(s) |
|---|---|
| Sender locks funds | `POST /v2/deposits` from sender's phone → poll until COMPLETED → insert `escrows` row with status `LOCKED` |
| Both parties confirm → release | `POST /v2/payouts` to recipient's phone → poll until COMPLETED → update `escrows.status = RELEASED` |
| Sender cancels before confirmation | `POST /v2/refunds` referencing the escrow's `depositId` → poll until COMPLETED → update `escrows.status = REFUNDED` |

## 9. Supabase — Direct Client Calls Only (No Edge Functions)

> **Decision made**: Edge functions were causing errors and burning time. For this hackathon, EVERYTHING talks to Supabase directly from the Expo client using `supabase-js`. Auth, database reads/writes, and realtime subscriptions all happen client-side. No edge functions, no custom backend. This is acceptable for a hackathon MVP — security hardening (RLS policies tightened, server-side validation, edge functions for sensitive operations) is a post-hackathon concern.

### 9.1 Supabase client setup

```js

  },
```

### 9.2 Auth — direct client calls

**Phone-based signup is the priority** since the app is phone-first. Supabase supports phone OTP auth directly — use it instead of building custom OTP logic.

```js
// Send OTP
async function sendOtp(phoneNumber) {
  const { data, error } = await supabase.auth.signInWithOtp({
    phone: phoneNumber, // E.164 format: +237653456789
  });
  if (error) throw error;
  return data;
}

// Verify OTP
async function verifyOtp(phoneNumber, token) {
  const { data, error } = await supabase.auth.verifyOtp({
    phone: phoneNumber,
    token,
    type: 'sms',
  });
  if (error) throw error;
  return data; // contains session + user
}

// Get current session
async function getSession() {
  const { data, error } = await supabase.auth.getSession();
  if (error) throw error;
  return data.session;
}

// Sign out
async function signOut() {
  const { error } = await supabase.auth.signOut();
  if (error) throw error;
}
```

> **Note**: Supabase phone OTP requires an SMS provider (Twilio, MessageBird, Vonage) configured in your Supabase project under Authentication → Providers → Phone. If this isn't configured and you're short on time, fall back to **email + password** auth instead — it works out of the box with zero config:

```js
// Fallback: email + password (zero config, works immediately)
async function signUp(email, password, fullName) {
  const { data, error } = await supabase.auth.signUp({
    email,
    password,
    options: { data: { full_name: fullName } },
  });
  if (error) throw error;
  return data;
}

async function signIn(email, password) {
  const { data, error } = await supabase.auth.signInWithPassword({ email, password });
  if (error) throw error;
  return data;
}
```

**Recommendation given your timeline**: use email + password for the demo if phone OTP isn't already configured and tested. Collect the phone number as a separate profile field used only for pawaPay calls — it does not need to be the auth identifier. This removes a dependency on SMS provider setup, which is a common time-sink.

### 9.3 Auth context (React)

```js
// /context/AuthContext.js
import { createContext, useContext, useEffect, useState } from 'react';
import { supabase } from '../lib/supabase/client';

const AuthContext = createContext(null);

export function AuthProvider({ children }) {
  const [session, setSession] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    supabase.auth.getSession().then(({ data }) => {
      setSession(data.session);
      setLoading(false);
    });

    const { data: listener } = supabase.auth.onAuthStateChange((_event, session) => {
      setSession(session);
    });

    return () => listener.subscription.unsubscribe();
  }, []);

  return (
    <AuthContext.Provider value={{ session, user: session?.user ?? null, loading }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () => useContext(AuthContext);
```

### 9.4 Database — direct CRUD from client

All table operations go straight through `supabase-js`. No edge functions wrapping these.

```js
// Create a circle
async function createCircle({ creatorId, name, goalType, circleType, targetAmount, contributionAmount, frequency }) {
  const { data, error } = await supabase
    .from('circles')
    .insert({
      creator_id: creatorId,
      name,
      goal_type: goalType,
      circle_type: circleType, // 'solo' | 'pool' | 'rotation'
      target_amount: targetAmount,
      contribution_amount: contributionAmount,
      frequency,
      invite_code: generateInviteCode(), // simple 6-char random string, client-generated
      status: 'active',
    })
    .select()
    .single();
  if (error) throw error;
  return data;
}

// Join a circle by invite code
async function joinCircle(inviteCode, userId) {
  const { data: circle, error: findError } = await supabase
    .from('circles')
    .select('*')
    .eq('invite_code', inviteCode)
    .single();
  if (findError) throw findError;

  const { data, error } = await supabase
    .from('circle_members')
    .insert({ circle_id: circle.id, user_id: userId, role: 'member', member_status: 'active' })
    .select()
    .single();
  if (error) throw error;
  return { circle, member: data };
}

// Record a contribution after pawaPay deposit completes
async function recordContribution({ circleId, memberId, pawapayDepositId, amount, cycleNumber }) {
  const { data, error } = await supabase
    .from('contributions')
    .insert({
      circle_id: circleId,
      member_id: memberId,
      pawapay_deposit_id: pawapayDepositId,
      amount,
      cycle_number: cycleNumber,
      status: 'PENDING', // update to COMPLETED after polling pawaPay
      due_at: new Date().toISOString(),
    })
    .select()
    .single();
  if (error) throw error;
  return data;
}

// Update contribution status after poll resolves
async function updateContributionStatus(contributionId, status) {
  const { data, error } = await supabase
    .from('contributions')
    .update({ status, paid_at: status === 'COMPLETED' ? new Date().toISOString() : null })
    .eq('id', contributionId)
    .select()
    .single();
  if (error) throw error;
  return data;
}

function generateInviteCode() {
  return Math.random().toString(36).substring(2, 8).toUpperCase();
}
```

### 9.5 Realtime — the "live dashboard" effect without webhooks

This is what replaces pawaPay webhooks/edge functions for the live feel. Subscribe to table changes directly from the client.

```js
// Subscribe to contribution updates for a specific circle (treasurer dashboard)
function subscribeToCircleActivity(circleId, onUpdate) {
  const channel = supabase
    .channel(`circle-${circleId}`)
    .on(
      'postgres_changes',
      {
        event: '*', // INSERT, UPDATE, DELETE
        schema: 'public',
        table: 'contributions',
        filter: `circle_id=eq.${circleId}`,
      },
      (payload) => onUpdate(payload)
    )
    .subscribe();

  return () => supabase.removeChannel(channel);
}

// Usage in a React component
useEffect(() => {
  const unsubscribe = subscribeToCircleActivity(circleId, (payload) => {
    console.log('Contribution updated:', payload);
    // refetch or merge into local state
  });
  return unsubscribe;
}, [circleId]);
```

This gives you the same "instant update on the dashboard" effect that webhooks would, but it's 100% client-driven — when your polling function updates a row via `updateContributionStatus`, every subscribed client sees it change in real time automatically.

### 9.6 Row Level Security — minimum viable security for the hackathon

Even going fully client-side, leave basic RLS on so users can't read/write each other's private data by accident. This takes 10 minutes and prevents embarrassing demo bugs (one user seeing another's wallet balance).

```sql
-- Run this in Supabase SQL editor for each table
alter table wallets enable row level security;
alter table circles enable row level security;
alter table circle_members enable row level security;
alter table contributions enable row level security;
alter table escrows enable row level security;

-- Wallets: users can only see/update their own
create policy "Users manage own wallet" on wallets
  for all using (auth.uid() = user_id);

-- Circles: anyone authenticated can read (needed to join via code), only creator can update
create policy "Anyone can read circles" on circles
  for select using (auth.role() = 'authenticated');
create policy "Creator can update own circle" on circles
  for update using (auth.uid() = creator_id);
create policy "Authenticated users can create circles" on circles
  for insert with check (auth.uid() = creator_id);

-- Circle members: members can read their circles, anyone can join
create policy "Members can read circle membership" on circle_members
  for select using (auth.role() = 'authenticated');
create policy "Users can join circles" on circle_members
  for insert with check (auth.uid() = user_id);

-- Contributions: members can read their circle's contributions
create policy "Members can read contributions" on contributions
  for select using (auth.role() = 'authenticated');
create policy "Users can insert own contributions" on contributions
  for insert with check (true); -- relaxed for hackathon speed

-- Escrows: only sender and recipient can see
create policy "Parties can view their escrow" on escrows
  for select using (auth.uid() = sender_id or auth.uid() = recipient_id);
create policy "Users can create escrows as sender" on escrows
  for insert with check (auth.uid() = sender_id);
```

> These policies are intentionally relaxed in places (e.g. contributions insert) to avoid debugging RLS errors mid-hackathon. Tighten after the deadline.

## 11. Critical Reminders for the Agent

1. **Generate UUIDs client-side** using `expo-crypto`'s `randomUUID()` or the `uuid` npm package — every `depositId`, `payoutId`, `refundId` must be a fresh UUIDv4 before calling the pawaPay API.
2. **Store the pawaPay ID immediately** in the relevant Supabase row (`contributions.pawapay_deposit_id`, `disbursements.pawapay_payout_id`, etc.) BEFORE waiting for the final result — this lets you recover if the app crashes mid-poll.
3. **Amount is always a string** in the pawaPay request body, never a number.
4. **Poll, don't assume** — `ACCEPTED` from the initiate call is NOT success. Only `COMPLETED` from the status check endpoint means the money actually moved.
5. **Use the Cameroon test numbers above for your live demo** — `237653456789` (MTN) and `237693456789` (Orange) for deposits and payouts that complete instantly and successfully.
6. **No edge functions anywhere — pawaPay or Supabase.** Every call is direct from the Expo client: `fetch()` straight to pawaPay, `supabase-js` straight to the database and auth.
7. **Use Supabase Realtime, not webhooks**, for the "live dashboard" feel — when your poll function updates a `contributions` row to `COMPLETED`, Supabase Realtime pushes that change to all subscribed clients instantly. This gives you the webhook-like live experience without needing a server to receive callbacks.
8. **If phone OTP auth isn't already working, switch to email + password immediately.** Don't lose hackathon time debugging SMS provider configuration. Phone number is still collected as a profile field for pawaPay calls — it just isn't the auth method.
9. **Keep RLS on with the relaxed policies provided** — don't disable RLS entirely, and don't spend time perfecting it either. The provided policies are the right level of effort for the time remaining.
10. **Test the full loop once, early** — create circle → contribute with demo number → poll resolves → Realtime updates dashboard → target reached → payout fires. Confirm this works end to end before building any more screens. This is the loop the judges will see.
